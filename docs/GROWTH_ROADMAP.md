# PlayPoint growth roadmap

## 目的

PlayPointの流入と継続利用を増やす。トップページへ機能を詰め込むのではなく、検索意図別の入口、計算結果画面、関連記事導線を強化して、利用者が「すぐ計算できる」「結果を保存・共有できる」「次に読むべき情報へ進める」状態を作る。

## 非目的

- トップページをポータル化しない。
- 記事やLPを無計画に量産しない。
- 通知、ランキング、バッジなどのゲーム化を先行しない。
- 広告枠を増やして短期収益を優先しない。
- ユーザー入力値、課金額、日記内容などをAnalyticsへ送らない。

## 基本方針

1. トップは計算体験を守る。
2. 追加機能は「結果画面」「個別LP」「記事導線」に逃がす。
3. Search Consoleの実クエリで優先順位を決める。
4. 新しい流入施策は、回帰テスト、sitemap、内部リンク、計測まで一緒に実装する。
5. デザインは情報量より判断しやすさを優先する。

## 成功条件

- Search Consoleで主要クエリのクリック数が増える。
- 計算完了イベントが増える。
- 記事から計算機への遷移が増える。
- 共有URL経由の流入を確認できる。
- トップページの初期表示と主要操作の使いやすさを悪化させない。

## 優先施策

### 1. 検索意図別LP

最優先で実装する。記事ではなく、計算機に直結する入口として作る。

候補:

- `/status/platinum/`
- `/status/diamond/`
- `/maintenance/platinum/`
- `/maintenance/diamond/`
- `/campaign/2x/`
- `/campaign/3x/`
- `/amount/10000/`

各LPに必要な要素:

- 1画面目に結論と計算フォームへの導線
- 対象ステータスや倍率を初期値にした計算リンク
- 使い方は短く、本文より計算導線を優先
- 関連記事は2から4件に絞る
- canonical、sitemap、内部リンク、構造化データを整備

やらないこと:

- 全金額、全倍率、全ステータスを一気に量産しない。
- 薄いテンプレページを増やさない。

受け入れ条件:

- 各LPから1クリックで条件付き計算を開始できる。
- sitemapに掲載される。
- 既存テストでリンク切れが検出される。
- Search Consoleで個別URLの表示回数を追える。

### 2. 計算結果画面の強化

トップに説明を足すのではなく、計算後の結果表示を強くする。

追加する表示:

- 必要額
- 月平均
- 週平均
- 1日あたり
- 年末までの残り日数
- 条件付き共有URL
- 関連記事導線

補助表示として検討:

- 目標達成の現実度
- 維持の危険度

デザイン方針:

- 結果はカードを増やしすぎず、主要値を3から4項目に絞る。
- 詳細値は折りたたみか補足行に置く。
- 広告より先に結果と次アクションを見せる。

受け入れ条件:

- 計算結果がスマホ幅で折り返しても読める。
- 結果表示が既存の通常計算、逆算、日記タブを邪魔しない。
- Analyticsへ具体的な課金額や必要ポイントを送らない。

### 3. 条件付き共有URLと共有文

既存の共有URLを成長施策として整える。

実装するもの:

- 結果画面に「この条件を共有」ボタン
- X向け共有文
- LINE向け共有文
- コピー用短文
- 記事から計算機へ条件付きで遷移するリンク

最初は画像生成をやらない。URL共有の利用が確認できた後に、画像生成を検討する。

受け入れ条件:

- 共有URLを開くと同じ条件が復元される。
- 不正なURLパラメータは無視される。
- 共有ボタンのタップ領域は44px以上。
- 共有イベントは個人情報なしで計測される。

### 4. 結果別関連記事導線

全記事を並べるのではなく、計算結果や入口ページに応じて少数だけ出す。

例:

- プラチナ目標: プラチナ維持、ランク維持、週次リワード
- ダイヤモンド目標: ダイヤモンド価値、VIP、課金計画
- 倍率あり: キャンペーン、ギフトカード割引
- 反映不安: 反映タイミング、反映されない原因

受け入れ条件:

- 表示件数は最大4件。
- 記事リンクは実在ファイルだけ。
- 計算結果より上に出さない。
- LP、記事、トップの導線が循環する。

### 5. Search Consoleベース改善

施策の優先順位はSearch Consoleで決める。

見る対象:

- 表示回数が多くCTRが低いクエリ
- 掲載順位8から20位のクエリ
- クエリはあるが対応ページが弱いもの
- 国別、言語別で伸びているクエリ

実装判断:

- 既存ページで答えられるならtitle、description、見出しを改善する。
- 意図が違うならLPを作る。
- 一時的なクエリなら記事にする。

受け入れ条件:

- 各改善に対象クエリを1つ以上紐づける。
- 改善後にSearch Consoleで確認する日付を決める。
- 変更内容をdocsまたはissue相当のメモに残す。

### 6. デプロイ後SEO Health Check

流入を増やす施策ではなく、流入を落とさないための防衛策として実施する。

実装するもの:

- Deploy workflow後にSEO Health Check相当の確認を実行
- title、description、canonical、robots、structured data、sitemap、記事広告タグを確認
- 失敗時はデプロイ成功と区別して原因が分かるログにする

受け入れ条件:

- デプロイ後に主要URLのSEO基本項目を確認できる。
- 週次SEO Health Checkと重複しすぎない。
- 本番の一時キャッシュで誤判定しにくい。

### 7. RSS/Atomフィード

ブログ記事を継続して増やすなら実装する。

受け入れ条件:

- 最新記事がフィードに入る。
- `head` からフィードを発見できる。
- sitemapやブログトップと記事一覧が矛盾しない。

## 段階的実装計画

### Phase 1: 土台

目的: これ以降の成長施策を壊さず測れる状態にする。

実装:

- Deploy後SEO Health Check統合
- Search Console改善メモの運用テンプレ
- 共有URLイベントの計測設計追加
- 結果別関連記事のデータ構造だけ定義

完了条件:

- CIでSEO基本項目が確認される。
- 追加イベントが個人情報を含まない。
- 既存テストが通る。

### Phase 2: 結果画面

目的: 既存トップのまま、計算後の満足度と共有率を上げる。

実装:

- 結果表示に月平均、週平均、1日あたりを追加
- 共有URLボタンを整理
- 結果別関連記事を最大4件表示

完了条件:

- スマホ幅で結果が読める。
- 計算完了から共有まで迷わない。
- 結果画面が広告や関連記事に押し負けない。

### Phase 3: 最初のLP

目的: SEO流入を増やす入口を作る。

実装:

- `/status/platinum/`
- `/status/diamond/`
- `/maintenance/platinum/`

完了条件:

- 各LPから条件付き計算へ遷移できる。
- sitemapに掲載される。
- トップ、関連記事、ブログから内部リンクされる。

### Phase 4: キャンペーン/金額LP

目的: 具体的な検索意図を拾う。

実装:

- `/campaign/2x/`
- `/campaign/3x/`
- `/amount/10000/`

完了条件:

- 倍率や金額が初期条件に反映される。
- 計算機側のURL復元テストが通る。

### Phase 5: 拡張判断

目的: 実データを見て増やすか止めるか判断する。

判断材料:

- Search Consoleの表示回数とクリック
- GA4の計算完了数
- 共有URL利用数
- 記事から計算機への遷移率
- LPごとの直帰に近い挙動

増やす候補:

- 多言語LP
- 結果画像生成
- 日記グラフ
- RSS/Atomフィード

## 今はやらない

- トップへの大量カード追加
- 全金額LPの一括生成
- 全倍率LPの一括生成
- ランキング、バッジ、連続記録などのゲーム化
- 強い通知導線
- 広告枠追加
- 記事の無計画な量産

## 実装時のチェックリスト

- トップページの主要導線を邪魔していない。
- スマホ幅で文字が詰まらない。
- 新規URLはsitemapに入っている。
- 内部リンクは実在ファイルを指している。
- Search Consoleで見る対象クエリがある。
- Analyticsイベントは個人情報を含まない。
- Service Workerやキャッシュバストの対象に入っている。
- 本番スモークまたはSEO Health Checkで検知できる。
