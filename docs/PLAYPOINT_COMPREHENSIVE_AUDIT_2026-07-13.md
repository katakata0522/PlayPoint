# Playポイント計算機 総合監査・改善報告

## 1. 結論・総合評価

ローカル監査の総合評価は **A-** です。計算機本体、生成物、記事の事実表現、SEO、アクセシビリティ、同意制御、公開準備用の回帰テストは大きく改善され、監査対象の主要フローは確認できました。

ただし、公開反映の判定は **HOLD** です。今回の作業では push／本番デプロイを行っていないため、公開中の `https://playpoint-sim.com/` はローカル修正前の状態です。特に、GitHub Actionsの手動ref制限とSVG出力修正は、公開・運用境界へ反映後にもう一度実サイト検証が必要です。

セキュリティ深掘りでは、報告対象は次の2件でした。

- **Medium:** `workflow_dispatch` で `main` 以外のrefを選んでも、本番SSH鍵とXserver rsyncへ到達できるCI認可欠落。
- **Low:** Kids Smile Landの証明書SVGで、子どもの名前がXMLエスケープなしにSVG本文へ入る問題。

詳細なCanonical Security ReportはTemp監査成果物として生成済みです。提案された最小修正は今回のローカル作業へ実装し、回帰テストも追加しました。

## 2. 監査範囲・方法

対象は `C:\Users\tomok\PlayPoint`、監査基準リビジョンは `e7abe0c24b37f5560051a5f73ac9ef2f3ef26632` です。45スロット、180作業行、7候補を確認し、各候補へ検証・攻撃経路分析を実施しました。最終的に2件が報告対象、5件はローカル限定の堅牢化またはサプライチェーン改善として整理されました。

確認した主な範囲は、計算ロジック、日記、記事と生成HTML、Service Worker、同意・GA・AdSense、埋め込み、Kindle Tracker、Kids Smile Land、Gravity TODO、楽天シミュレーター、サブスク健康診断、GitHub Actions、Xserver配信境界です。

## 4. 実装した改善

主な変更は次のとおりです。

- `deploy.yml` に `github.ref == 'refs/heads/main'` のjob-levelガードを追加。
- Kids Smile LandのSVG本文へXMLエスケープを追加し、入力・保存・ファイル名を分離。
- Kindle Trackerのバックアップを5MB、1,000件で早期拒否。
- Gravity TODOの保存復元を100件で制限。
- 楽天シミュレーターの保存配列と価格を正規化。
- サブスク診断のカスタム保存を100件で制限。
- 同意バナー表示直後の同期focusを次フレームへ移し、強制レイアウトを緩和。
- 小数のHTML `step`違反を計算入力で拒否。
- 言語切替後の記事件数を実数で再描画。
- フッターリンクの下線、モバイルヘッダーのbox-sizingを改善。
- 公式情報と整合しない、または現在性を確認できない記事表現を修正。
- 生成HTML、ブログ一覧、RSS／Atom、sitemap、Service Workerの版番号・日付を再生成。

## 7. SEO・アクセシビリティ・パフォーマンス

ローカルLighthouseは、Performance **78**、Accessibility **100**、Best Practices **100**、SEO **100**でした。FCP 2.9秒、LCP 3.2秒、TBT 370ms、CLS 0、Speed Index 3.5秒です。静的サーバー・未minifyソースを対象にした値なので、本番CDN・gzip・キャッシュ条件とは分けて扱います。

公開サイトの既存スナップショットはPerformance 78、Accessibility 97、Best Practices 100、SEO 100でした。公開側には旧CSS由来のリンク識別、初期描画、JS minify・cache lifetimeの改善余地が残っており、今回のローカル修正がまだ公開へ反映されていないことと整合します。

SEOは、canonical／hreflang／sitemap／RSS／Atom／JSON-LD／記事導線を確認済みです。改善の重点は、外部キャンペーンの現在性と、記事更新日・本文・構造化データの三者同期を崩さない運用です。

## 8. トラフィック・Analytics

この監査では、Google Search Console、GA4の実数、広告収益、検索クエリ、順位、インデックス状況の実データは取得できませんでした。したがって、流入や収益を推測して評価していません。

実装面では、GA4とAdSenseは同意後ロード、計算完了・日記保存・記事導線・LP流入のイベント設計を確認し、個人情報をイベントへ入れない回帰テストが通過しています。次回の運用監査では、Search Consoleのクリック・表示・CTR・掲載順位と、GA4の同意率・計算完了率・記事→計算機遷移率を月次で保存してください。

## 9. マネタイズ評価

`ads.txt` は存在し、AdSense publisher recordと一致しました。広告・Analyticsの同意制御も通過しています。一方、ローカルコード上の楽天・サブスク関連アフィリエイトIDには `YOUR_AFFILIATE_ID` 系のプレースホルダーが残っているため、アフィリエイト収益が本番で有効だとは判定していません。

収益化の優先順位は、広告を増やすことよりも、検索意図別LPから計算完了へ自然に移動できること、記事の現在性を保つこと、公式情報との不一致を防ぐことです。キャンペーン記事は、掲載前に対象国・期間・対象商品・還元上限・終了済み表示を毎回確認する運用が必要です。
